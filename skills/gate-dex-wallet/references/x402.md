# Gate DEX x402 Payment

> x402 payment module — pay for APIs that return `402 Payment Required` using the wallet. **Two-step wallet tools:** **`dex_tx_x402_checkin_preview`** (no signing; prepares Gate Verify `message` for **EVM exact / EIP-3009**) and **`dex_tx_x402_fetch`** (signs after check-in, retries with `PAYMENT-SIGNATURE`). **Do not** have the agent compute EIP-3009 digests by hand — use the preview tool.

## Check-in before x402 (mandatory)

**Order is fixed:** for **EVM exact / EIP-3009** (typical CoinGecko Pro x402 on Base): **`dex_tx_x402_checkin_preview`** → Gate Verify **`/v1/tx/checkin`** / **`tx_checkin`** ([tx-checkin.md](./tx-checkin.md)) → **`dex_tx_x402_fetch`**. **Do not** call `dex_tx_x402_fetch` first to probe 402.

**Check-in MCP auth:** call **`/v1/tx/checkin`** / **`tx_checkin`** with **`authorization`** in tool **arguments** set to the **same `mcp_token`** as wallet MCP tools — **not** by duplicating the wallet server’s HTTP **`headers`** on the Verify MCP config — see [tx-checkin.md](./tx-checkin.md). If you debug Verify via raw HTTP, some gateways need **`Accept: application/json, text/event-stream`**; MCP clients normally handle transport headers separately.

### EVM exact / EIP-3009 (recommended path)

1. Confirm payment intent when needed (see **Execution Flow**).
2. Call **`dex_tx_x402_checkin_preview`** with the same `url` / `method` / `body` / `headers` / `x402_preferred_network` you will use for fetch. Optionally pass **`x402_payment_required_b64`**; if omitted, the tool performs **one** outbound HTTP call and requires **HTTP 402** with a `PAYMENT-REQUIRED` header.
3. From the result, read **`tx_checkin`** (object with `wallet_address`, `type`, `source`, `message`, `chain`, `chain_category`) and **`x402_payment_required_b64`**. Call Gate Verify **`tx_checkin`** / **`/v1/tx/checkin`** with **`authorization`** (same **`mcp_token`** as wallet) **plus** the **`tx_checkin`** fields as arguments — the **`message`** is the 64-hex EIP-3009 digest; **do not** substitute a self-computed digest.
4. On Verify success, read **`checkin_token`** from **`data.checkin_token`**.
5. Call **`dex_tx_x402_fetch`** with **`checkin_token`**, the **same** `url` / `method` / `body` / `headers` as preview, and the **same** **`x402_payment_required_b64`** string returned by the preview (required for digest/nonce alignment when supplied).

If preview or check-in fails, **abort** — do not proceed to `dex_tx_x402_fetch`.

### Solana or EVM upto (Permit2)

**`dex_tx_x402_checkin_preview` currently supports only EVM `scheme=exact` (EIP-3009).** For Solana or Permit2 **upto**, follow product / backend rules for **`tx_checkin`** `message` / `intent` (preview may return an error — do not fabricate digests). Then call **`dex_tx_x402_fetch`** with **`checkin_token`** as today.

## Applicable Scenarios

Use when the user wants to pay for a 402 resource or call an API that requires x402 payment:
- Explicit payment language: "pay for this API", "402 payment", "x402 pay", "payment required"
- Fetch-and-pay: "call this URL and pay if 402", "pay and fetch", "x402 fetch"
- URL that returns 402: user provides a URL (e.g., flight order, gated API, premium content)
- Access-gated resources: "fetch this paid endpoint", "pay for access", "unlock this resource"
- Retry after 402: "that URL returned 402, can you pay for it?"

## Capability Boundaries

- Supports: Fetching URLs with automatic x402 payment, EVM exact (EIP-3009), EVM upto (Permit2), Solana exact (SPL), Solana upto (SPL)
- Supported chains: Any EIP-155 EVM chain (eth, base, bsc, etc.) and Solana mainnet/devnet
- Does **not** support: Token transfers (-> [transfer.md](./transfer.md)), token swaps (-> `gate-dex-trade`), DApp interactions (-> [dapp.md](./dapp.md)), market data queries (-> `gate-dex-market`)

**Prerequisites**: MCP Server connected (see parent SKILL.md). User must be logged in (`mcp_token`). Wallet must have an EVM address and/or Solana address for the chain the 402 server accepts.

**Signing / check-in**: Parent [SKILL.md](../SKILL.md) requires Gate Verify [tx-checkin.md](./tx-checkin.md) **before** every `dex_tx_x402_fetch` call. For **EVM EIP-3009**, use **`dex_tx_x402_checkin_preview`** first so **`message`** matches the digest that will be signed. The agent **must** pass **`checkin_token`** from Verify into `dex_tx_x402_fetch`, and when using preview for EIP-3009, pass the **same** **`x402_payment_required_b64`** from the preview result.

### Design notes (EIP-3009 digest, `PAYMENT-REQUIRED` b64, HTTP 415)

**Why `tx_checkin.message` is 64-hex, not the URL** — Gate Verify registers the **exact signing payload**. For x402 **EVM `exact` + EIP-3009**, the wallet signs the **EIP-712 digest** for `transferWithAuthorization` (32 bytes → 64 hex). That digest is bound to the chosen `accept` and the **raw** `PAYMENT-REQUIRED` header (base64). A URL string is not a valid `message` for this path and would not match the later `PAYMENT-SIGNATURE`. **`dex_tx_x402_checkin_preview`** exists so agents never hand-build the digest.

**Probe vs caller-supplied `x402_payment_required_b64`** — Omitting it triggers **one** outbound HTTP call that must return **402** plus the header. Passing it **skips** that call; the b64 must be **byte-identical** to the header used when the digest was computed (use the preview’s returned `x402_payment_required_b64` in the same flow as `dex_tx_x402_fetch`). These are two ways to obtain the same snapshot, not mutually exclusive logic.

**POST JSON and `415 Unsupported Media Type`** — If `method` is `POST` and `body` is JSON, **always** pass `headers` including **`"Content-Type":"application/json"`**. The wallet MCP does not infer it; missing `Content-Type` often yields **415** on both the preview’s probe and the paid retry (retry **clones** your headers and adds payment headers). Use the **same** `url` / `method` / `body` / `headers` JSON string for **`dex_tx_x402_checkin_preview`** and **`dex_tx_x402_fetch`**. If 415 persists with correct `Content-Type`, the resource gateway may reject the payment retry headers or body rules — treat as upstream/MCP compatibility, not as “digest vs b64 deadlock.”

## MCP Tools

### 1. `dex_tx_x402_checkin_preview` — Prepare `tx_checkin` for EVM EIP-3009 (no signing)

Builds the Gate Verify **`tx_checkin`** arguments and the raw **`x402_payment_required_b64`** for **EVM `scheme=exact` (EIP-3009)** without signing. **Does not** replace Gate Verify — call **`tx_checkin`** immediately after with the returned `tx_checkin` object.

| Field | Description |
|-------|-------------|
| **Parameters** | `{ url, method?, body?, headers?, mcp_token?, x402_preferred_network?, x402_payment_required_b64? }` — same shape as fetch for the outbound probe when `x402_payment_required_b64` is omitted. |
| **Returns** | `{ tx_checkin: { wallet_address, type, source, message, chain?, chain_category? }, x402_payment_required_b64, scheme, network, sign_method }` — use `tx_checkin.*` as Verify arguments; keep `x402_payment_required_b64` for **`dex_tx_x402_fetch`**. |

**Scope:** If the selected accept is **Solana** or **EVM upto**, the tool returns an explicit error — follow **Solana or EVM upto** above; do not guess digests.

#### Call Example (CoinGecko-style GET)

```text
CallMcpTool(
  server="<wallet-mcp-server>",
  toolName="dex_tx_x402_checkin_preview",
  arguments={
    url: "https://pro-api.coingecko.com/api/v3/x402/simple/price?vs_currencies=usd&symbols=btc,eth,sol"
  }
)
# Then: tx_checkin with arguments = { authorization: <same mcp_token as wallet>, ...result.tx_checkin } (see tx-checkin.md)
# Then: dex_tx_x402_fetch with url, checkin_token, x402_payment_required_b64 = result.x402_payment_required_b64
```

### 2. `dex_tx_x402_fetch` — Request URL with Automatic x402 Payment

Send an HTTP request to the given URL. If the server responds with 402 Payment Required, pay via x402 using the current user's wallet (EVM: EIP-3009 exact, Permit2 upto; Solana: SPL exact/upto), retry automatically, and return the final response or an error.

| Field | Description |
|-------|-------------|
| **Parameters** | `{ url, method?, body?, headers?, mcp_token?, checkin_token?, x402_payment_required_b64?, x402_preferred_network? }` — **`checkin_token`** required when the tool defines it. After **`dex_tx_x402_checkin_preview`** for EIP-3009, pass the **same** **`x402_payment_required_b64`** as preview. |
| **Returns** | `{ status_code, body, headers? }` of the **final** response after any 402 -> pay -> retry |

#### Parameter Details

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `url` | string | Yes | Full request URL (e.g., `https://webws.gate.io:443/flight/order`) |
| `method` | string | No | HTTP method; default `GET` |
| `body` | string | No | Request body (e.g., JSON string for POST) |
| `headers` | string | No | JSON object of headers, e.g., `{"Content-Type":"application/json"}` |
| `mcp_token` | string | No | Auth token; optional if Authorization header is set by client |
| `checkin_token` | string | When required by gateway / tool | From successful **`/v1/tx/checkin`** response **`data.checkin_token`**; **always** run check-in **before** this tool (see **Check-in before x402**). |
| `x402_payment_required_b64` | string | No | When set, skips the first HTTP call; **must** match the EIP-3009 digest used in `tx_checkin` (use the string from **`dex_tx_x402_checkin_preview`** for CoinGecko-style flows). |
| `x402_preferred_network` | string | No | e.g. `base`, `solana`, `eip155:8453` — filters `accepts`; CoinGecko `/x402/` defaults to Base when omitted. |

#### Call Example (POST with JSON body)

```text
CallMcpTool(
  server="<wallet-mcp-server>",
  toolName="dex_tx_x402_fetch",
  arguments={
    url: "https://example.com/api/order",
    method: "POST",
    body: "{\"flightId\":\"FL001\",\"uid\":\"100\"}",
    headers: "{\"Content-Type\":\"application/json\"}",
    checkin_token: "<from_verify_mcp_data_checkin_token>"
  }
)
```

#### Response Example (success after 402 -> pay -> 200)

```json
{
  "status_code": 200,
  "body": "{\"code\":\"0\",\"message\":\"success\",\"data\":{\"orderId\":\"ORD-xxx\",\"status\":\"PAID\",\"orderAmount\":\"0.02\",\"network\":\"eth\",\"asset\":\"0xA0b8...\",\"payTo\":\"0x...\"}}",
  "headers": { "Content-Type": ["application/json"] }
}
```

Agent behavior: Parse `body` and show user **orderId**, **status** (PAID), **amount**, **chain/network**. If `status_code` is not 2xx, show error from body or status.

## Supported Payment Types

| Type | Chain / Token | Note |
|------|---------------|------|
| **EVM exact** | Any EIP-155 chain (eth, base, etc.) + EIP-3009 token (e.g., USDC) | User must have EVM address and token balance |
| **EVM upto** | Same chains + ERC-20 via **Permit2** + x402 upto proxy | `amount` = **maximum** authorized; 402 `accepts[].extra` must include `facilitator` (EVM address that will call `settle`). User needs one-time ERC-20 `approve(Permit2)` if not already approved. |
| **Solana exact** | Solana mainnet/devnet + SPL token | User must have Solana address and token balance |
| **Solana upto** | Same as Solana exact at signing time | Client signs SPL transfer for **max** `amount`; facilitator settles actual usage per server (spec still evolving). |

### Payment Scheme Selection Priority

The server's 402 response lists **accepts** (e.g., `scheme`, `network`, `asset`, `payTo`, `amount`, `extra`). The tool selects in order:

**Solana exact -> EVM exact -> Solana upto -> EVM upto**

matching the user's wallet addresses and token balances, then builds the payment payload and retries with `PAYMENT-SIGNATURE`.

### Transaction Recording

When payment succeeds, the backend may write a row to `wallet_transactions` (e.g., `business_type='x402_payment'`). No extra user action is needed.

## Execution Flow

```
Step 0: MCP Server connection detection (once per session)
  |
Step 1: Authentication check
  |- No mcp_token -> Route to auth.md
  +- Valid token  -> Continue
  |
Step 2: Confirm payment intent
  |- If user only pastes a URL without explicit payment language,
  |  confirm: "This URL may require payment (402). Proceed?"
  +- If user explicitly says "pay" / "x402" / "402" -> proceed directly
  |
Step 2b: EVM EIP-3009 — `dex_tx_x402_checkin_preview` (mandatory before tx_checkin for CoinGecko-style exact)
  |- Same url/method/body/headers (and optional x402_preferred_network) as fetch will use
  |- Capture tx_checkin object and x402_payment_required_b64
  +- On failure -> abort (or follow Solana/upto branch per doc)
  |
Step 2c: Gate Verify MCP `/v1/tx/checkin` (mandatory before fetch)
  |- For EIP-3009: pass fields from preview.tx_checkin as Verify arguments
  |- On success, capture checkin_token from data.checkin_token
  +- On failure -> abort; do not call dex_tx_x402_fetch
  |
Step 3: Call dex_tx_x402_fetch({ url, method?, body?, headers?, mcp_token?, checkin_token, x402_payment_required_b64? })
  |- EIP-3009: x402_payment_required_b64 must match preview output byte-for-byte
  |
Step 4: Server response handling
  |- 200/2xx (no 402) -> Return response to user (no payment occurred)
  |- 402 with x402 payload -> Tool auto-pays and retries:
  |   |- Selects payment scheme in priority order:
  |   |  Solana exact -> EVM exact -> Solana upto -> EVM upto
  |   |- For exact: builds EIP-3009 signature (EVM) or SPL transfer (Solana)
  |   |- For upto: builds Permit2 signature (EVM) or SPL max-transfer (Solana)
  |   |  (EVM upto requires `facilitator` in accepts[].extra)
  |   |- Signs payment, retries with PAYMENT-SIGNATURE header
  |   +- Returns final response (usually 200 + order/PAID status)
  |
Step 5: Display result
  |- Parse body for orderId, status (PAID), amount, chain/network
  +- If not 2xx, show error from body or status
```

## Conversation Examples

**Example 1 (Happy Path): Pay for a gated API — GET request (EVM EIP-3009)**
User: "Call https://example.com/api/order and pay if needed"
Agent:
1. Verify `mcp_token` is valid.
2. Call **`dex_tx_x402_checkin_preview`** with the same URL (and method/headers) as fetch; capture **`tx_checkin`** and **`x402_payment_required_b64`** (or handle preview error if not EIP-3009 exact).
3. Call Gate Verify **`/v1/tx/checkin`** with the **`tx_checkin`** fields; capture **`checkin_token`** from **`data.checkin_token`**.
4. Call `dex_tx_x402_fetch({ url: "...", checkin_token, x402_payment_required_b64 })` (same URL as preview; **same** b64 when preview returned one).
5. If 402 -> tool auto-pays -> display final response (orderId, status, amount paid).

**Example 2 (Happy Path): POST with JSON body (EVM EIP-3009, e.g. flight order)**
User: "Submit a flight order to https://webws.gate.io/flight/order with body {flightId: FL001}"
Agent:
1. Verify `mcp_token` is valid.
2. Call **`dex_tx_x402_checkin_preview`** with `method: "POST"`, JSON `body`, and `headers: "{\"Content-Type\":\"application/json\"}"` (same triple you will use for fetch); capture **`tx_checkin`** and **`x402_payment_required_b64`**.
3. Call Gate Verify **`tx_checkin`** with **`authorization`** plus the **`tx_checkin`** fields from step 2; capture **`checkin_token`**.
4. Call `dex_tx_x402_fetch` with the **same** `url` / `method` / `body` / `headers`, **`checkin_token`**, and **`x402_payment_required_b64`** from step 2.
5. Display result (orderId, status, amount, tx explorer if present).

**Example 3 (Happy Path): No payment needed — public API**
User: "Fetch https://example.com/public-api"
Agent:
1. Call **`/v1/tx/checkin`**; capture **`checkin_token`** from **`data.checkin_token`** (required before the tool even if no 402).
2. Call `dex_tx_x402_fetch({ url: "...", checkin_token })`.
3. Server returns 200 -> display response directly (no payment occurred).

**Example 4 (Happy Path): Explicit x402 payment**
User: "Use x402 to pay for https://premium.api.io/data"
Agent:
1. User explicitly says "x402" — proceed directly without additional confirmation.
2. For **EVM EIP-3009**: **`dex_tx_x402_checkin_preview`** → **`tx_checkin`** → **`dex_tx_x402_fetch`** (with **`x402_payment_required_b64`** from preview when returned). For **Solana / EVM upto** where preview is unsupported, follow [Solana or EVM upto](#solana-or-evm-upto-permit2) then **`tx_checkin`** then **`dex_tx_x402_fetch`** per product rules.
3. Display payment result.

**Example 5 (Edge Case): URL pasted without explicit payment intent**
User: "https://example.com/paid-endpoint"
Agent:
1. User only pasted a URL without saying "pay."
2. Confirm: "This URL may require payment (HTTP 402). Would you like me to fetch it and pay if needed?"
3. On user confirmation -> for **EIP-3009**: **`dex_tx_x402_checkin_preview`** → **`tx_checkin`** → **`dex_tx_x402_fetch`**; otherwise **`tx_checkin`** then **`dex_tx_x402_fetch`** per [Solana or EVM upto](#solana-or-evm-upto-permit2).

**Example 6 (Edge Case): Insufficient balance for payment**
User: "Pay for https://example.com/api/premium"
Agent:
1. Run the correct flow for the scheme (**EIP-3009**: preview → **`tx_checkin`** → fetch with **`x402_payment_required_b64`**; otherwise per upto/Solana rules), then call **`dex_tx_x402_fetch`** with **`checkin_token`**.
2. Tool returns error: "Request or payment failed: insufficient USDC balance."
3. Display: "Payment failed — insufficient USDC balance. The resource requires 0.50 USDC but your wallet has 0.10 USDC. Please top up and retry."

**Example 7 (Edge Case): 402 without x402 protocol**
User: "Fetch this URL: https://legacy-api.com/resource"
Agent:
1. Call **`/v1/tx/checkin`**, then call `dex_tx_x402_fetch` with **`checkin_token`**.
2. Server returns 402 but without x402 payload.
3. Display: "The server returned 402 Payment Required but does not follow the x402 protocol. Automatic payment is not possible."

**Example 8 (Edge Case): Retry a previously paid request**
User: "Call that same URL again"
Agent:
1. Warn: "Retrying may result in a duplicate payment if the server doesn't deduplicate by order ID. Proceed?"

**Example 9 (Edge Case): Custom headers needed**
User: "Call https://api.io/order with header Authorization: Bearer mytoken123"
Agent:
1. For **POST JSON**, merge custom headers with **`Content-Type: application/json`** in one `headers` JSON string.
2. **EIP-3009**: **`dex_tx_x402_checkin_preview`** and **`dex_tx_x402_fetch`** must use the **same** `headers` string; then **`tx_checkin`** with preview’s **`tx_checkin`**, then fetch with **`checkin_token`** and **`x402_payment_required_b64`** when applicable.
3. Display result.

**Example 10 (Edge Case): EVM upto payment with Permit2**
User: "Pay for https://premium.api.io/data — the server uses Permit2 upto"
Agent:
1. Verify `mcp_token` is valid.
2. Call **`/v1/tx/checkin`**; capture **`checkin_token`** from **`data.checkin_token`**.

4. Tool selects EVM upto scheme (if `facilitator` is present in `accepts[].extra`), builds Permit2 signature, retries.
5. If user has not approved the Permit2 contract for the token, tool returns a Permit2 allowance error — guide user to approve Permit2 first via [dapp.md](./dapp.md).
6. After approval, **`/v1/tx/checkin`** again and retry `dex_tx_x402_fetch` with a fresh **`checkin_token`**.

**Example 11 (Edge Case): Permit2 approval needed before upto payment**
User: "That payment failed with a Permit2 allowance error"
Agent:
1. The token has not been approved for the Permit2 contract.
2. Guide user: "You need to approve the Permit2 contract for this token first. Would you like me to help with that?"
3. On confirmation, route to [dapp.md](./dapp.md) for ERC-20 `approve(Permit2)`.
4. After approval, call **`/v1/tx/checkin`** again and retry `dex_tx_x402_fetch` with **`checkin_token`**.

**Example 12 (Edge Case): Server missing facilitator for upto**
User: "Pay for https://api.io/resource"

2. Tool returns error about missing `facilitator` in `accepts[].extra`.
3. Display: "The server requires upto payment but did not include a facilitator address in its 402 response. This is a server-side configuration issue. Please contact the API provider."

## Error Handling

| Scenario | Handling |
|----------|----------|
| No `mcp_token` or auth failed | Route to [auth.md](./auth.md) for login |
| "Failed to retrieve EVM or Solana address" | Tell user to bind EVM or Solana address for the wallet |
| "no supported payment option" / unsupported scheme | The 402 `accepts` are not compatible with the user's wallet; inform user |
| EVM upto error about `facilitator` | Resource server must include `facilitator` address in `accepts[].extra`; inform user the server configuration is incomplete |
| EVM upto / Permit2 "allowance" or precondition errors | User may need to approve the canonical Permit2 contract once for that token; guide to [dapp.md](./dapp.md) for ERC-20 `approve(Permit2)` |
| "Request or payment failed: ..." | Display error message; may indicate insufficient balance, wrong chain, or server error |
| `status_code` 4xx/5xx in final response | Display status and body to user. **415**: first ensure POST JSON used `headers` with **`Content-Type: application/json`** on **both** preview and fetch; if still 415, resource server may reject the payment retry — escalate to provider/MCP compatibility |
| Timeout / DNS error calling target URL | Prompt user to verify the URL is correct and reachable |
| 402 response without x402 payload | Inform user the server returned 402 but does not follow x402 protocol; cannot auto-pay |
| `dex_tx_x402_checkin_preview` error (Solana / upto / not 402) | Follow **Solana or EVM upto** section; do not invent digests. If probe expected 402, confirm URL and that resource uses x402 |
| Facilitator / `Invalid payment` after signed retry | Show server message; user may need correct chain balance, or provider-side facilitator issue — not fixed by repeating preview alone |

## Security Rules

1. **Check-in first**: Never call `dex_tx_x402_fetch` before a successful **`/v1/tx/checkin`** in the same flow; never use the fetch tool to probe whether check-in is required.
2. **EVM EIP-3009 digest**: Never ask the model to derive the 64-hex EIP-3009 digest manually — use **`dex_tx_x402_checkin_preview`** and pass its **`tx_checkin.message`** to Verify, then pass the same **`x402_payment_required_b64`** to **`dex_tx_x402_fetch`**.
3. **Confirm intent**: If the user only pastes a URL without explicit payment language, confirm they want to pay if the server asks for payment (402). Never auto-pay without user awareness.
4. **Token confidentiality**: Never log or display `mcp_token` in full. Use placeholders like `<mcp_token>`.
5. **Amount transparency**: If the 402 response or final 200 body includes amount/asset, display them so the user knows what was (or will be) paid. For **upto** schemes, clarify that the amount is the **maximum** authorized — actual settlement may be less.
6. **Idempotency warning**: Same URL/body can be retried; duplicate payments may occur if the server does not deduplicate by id/orderId. Warn the user before retrying a previously paid request.
7. **No data fabrication**: Report all responses truthfully. Never invent payment results, amounts, or order IDs.
8. **URL validation**: Only send requests to user-provided URLs. Never construct or modify payment URLs autonomously.
9. **Permit2 awareness**: For EVM upto payments, the user may need a one-time ERC-20 `approve(Permit2)`. If the tool reports allowance errors, guide the user through the approval — do not silently skip.
