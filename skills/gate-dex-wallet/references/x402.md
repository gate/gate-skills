# Gate DEX Wallet — x402 Payment

> **HTTP 402 Payment** — Pay for APIs that return `402 Payment Required` using the wallet. Tool **`dex_tx_x402_fetch`**: send `url` plus optional `method` / `body` / `headers`; on 402 with x402 `accepts`, the MCP server signs the chosen scheme and retries with `PAYMENT-SIGNATURE`.

**Trigger scenarios**: User wants to **pay for a 402 resource** or **call an API that requires x402 payment**:
- "pay for this API", "402 payment", "x402 pay", "payment required"
- "call this URL and pay if 402", "pay and fetch", "x402 fetch"
- User provides a URL that returns 402 (e.g. flight order, gated API)

**NOT this module** (common misroutes):
- "transfer ETH to 0xABC" → `references/transfer.md` (token transfer)
- "swap ETH for USDT" → `gate-dex-trade` (swap execution)
- "check my balance" → Parent SKILL.md (asset query)

**Prerequisites**: MCP Server connected (see parent SKILL.md). User must be **logged in** (`mcp_token`). Wallet must have an **EVM address** and/or **Solana address** (for the chain the 402 server accepts).

## Product Definition (Knowledge Layer)

x402 is an HTTP-based payment protocol. When a server returns **HTTP 402 Payment Required**, the response body contains an x402 payload with one or more `accepts` entries specifying acceptable payment methods (chain, token, amount, payee). The wallet client selects a compatible method, signs the payment, and retries the original request with a `PAYMENT-SIGNATURE` header. The server verifies payment and returns the resource.

### Supported Payment Schemes

| Scheme | Chain / Token | How It Works |
|--------|---------------|--------------|
| **EVM exact** | Any EIP-155 chain (eth, base, etc.) + EIP-3009 token (e.g. USDC) | Signs exact amount transfer via EIP-3009; user must have EVM address and token balance |
| **EVM upto** | Same chains + ERC-20 via **Permit2** | Signs permit for **maximum** `amount`; requires `extra.facilitator` address in 402 payload; user may need one-time `approve(Permit2)` |
| **Solana exact** | Solana mainnet / devnet + SPL token | Signs exact SPL transfer; user must have Solana address and token balance |
| **Solana upto** | Same as Solana exact | Signs SPL transfer for **max** `amount`; facilitator settles actual usage (spec evolving) |

### Selection Priority

When the 402 response lists multiple `accepts`, the tool picks in order: **Solana exact → EVM exact → Solana upto → EVM upto**, matching the user's wallet addresses.

## Scenarios and Boundaries (Decision Layer)

### Applicable Scenarios

- User wants to pay for a 402-gated API endpoint
- User provides a URL and asks to "pay if needed"
- User wants to call a resource that requires x402 payment (e.g. flight booking, premium API)

### Capability Boundaries

| Supported | Not Supported |
|-----------|---------------|
| Pay for HTTP 402 resources via x402 protocol | Regular token transfers (→ `references/transfer.md`) |
| Automatic scheme selection and signing | Manual payment construction |
| GET/POST/PUT/DELETE with auto-pay on 402 | Non-HTTP payment protocols |
| EVM (EIP-3009, Permit2) and Solana (SPL) schemes | Bitcoin or other non-EVM/non-Solana chains |

## Execution Spec (Execution Layer)

### x402 Flow

1. **Auth**: Ensure `mcp_token` is valid; if not, route to `references/auth.md`.
2. **Confirm intent**: If the user only pastes a URL without explicit pay intent, confirm they want to pay if the server asks for payment (402).
3. **Request**: Call **`dex_tx_x402_fetch`** with `url`, and optionally `method`, `body`, `headers`, `mcp_token`.
4. **Server behavior**:
   - If server returns **200/2xx** → return response to user (no payment needed).
   - If server returns **402** with x402 payload → the MCP wallet client selects one `accept` in priority order, builds the payload, signs, and **retries** with `PAYMENT-SIGNATURE`; then returns the final response.
5. **Display result**: Parse response `body` and show user key fields (**orderId**, **status**, **amount**, **chain/network**).

### MCP Tool: `dex_tx_x402_fetch`

| Field | Description |
|-------|-------------|
| **Tool Name** | `dex_tx_x402_fetch` |
| **Parameters** | See table below |
| **Return** | `{ status_code, body, headers? }` for the **final** response after any 402 → pay → retry |

#### Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `url` | string | Yes | Full request URL (e.g. `https://webws.gate.io:443/flight/order`) |
| `method` | string | No | HTTP method; default `GET` |
| `body` | string | No | Request body, e.g. JSON string for POST |
| `headers` | string | No | JSON object string of headers, e.g. `{"Content-Type":"application/json"}` |
| `mcp_token` | string | No | Auth token; optional if the client sends `Authorization` |

#### Call Example (POST with JSON body)

```text
CallMcpTool(
  server="<wallet-mcp-server>",
  toolName="dex_tx_x402_fetch",
  arguments={
    url: "https://example.com/api/order",
    method: "POST",
    body: "{\"flightId\":\"FL001\",\"uid\":\"100\"}",
    headers: "{\"Content-Type\":\"application/json\"}"
  }
)
```

#### Response Example (success after 402 → pay → 200)

```json
{
  "status_code": 200,
  "body": "{\"code\":\"0\",\"message\":\"success\",\"data\":{\"orderId\":\"ORD-xxx\",\"status\":\"PAID\",\"orderAmount\":\"0.02\",\"network\":\"eth\",\"asset\":\"0xA0b8...\",\"payTo\":\"0x...\"}}",
  "headers": { "Content-Type": ["application/json"], "Date": ["..."] }
}
```

Agent behavior: Parse `body` and show user **orderId**, **status** (PAID), **amount**, **chain/network**. If `status_code` is not 2xx, show error from body or status.

## Business-Specific Error Handling

| Situation | Agent Action |
|-----------|-------------|
| No `mcp_token` or auth failed | Route to `references/auth.md` for login |
| Error: "未获取到 EVM 或 Solana 地址" (no EVM/Solana address found) | Tell user to bind an EVM or Solana address in their wallet before retrying |
| Error: "no supported payment option" / unsupported scheme | The 402 `accepts` are not compatible with user's wallet; inform user which chains/tokens are required |
| EVM upto: missing `facilitator` | The resource server's 402 response must include `extra.facilitator`; this is a server-side issue — inform user |
| EVM upto: Permit2 allowance error | User needs to approve the canonical **Permit2** contract once for that token; guide them to approve via DApp module → `references/dapp.md` |
| Error: "请求或支付失败: ..." (request or payment failed) | Show the error detail; likely causes: insufficient balance, wrong chain, or server error |

## Security and UX

1. **Confirm intent**: If the user only pastes a URL, confirm they want to **pay** if the server asks for payment (402).
2. **Sensitive data**: Do not log or display `mcp_token` in full.
3. **Amount visibility**: If the 402 response or final 200 body includes amount/asset, always show them so the user knows what was (or will be) paid.
4. **Idempotency warning**: Same URL/body can be retried; duplicate payments may occur if the server does not deduplicate by id/orderId.

## Conversation Examples

### Example 1: Normal x402 Payment (Happy Path)

**User**: "Pay for this API: https://example.com/api/flight/order"
**Agent**:
1. Identify intent: x402 payment, url provided, GET method (default)
2. Verify `mcp_token` is valid
3. Call `dex_tx_x402_fetch` with `url: "https://example.com/api/flight/order"`
4. Tool returns `status_code: 200` with body containing orderId, status: PAID, amount: 0.02 USDC on eth
5. Display to user: "Payment successful! Order ORD-xxx confirmed. Paid 0.02 USDC on Ethereum."

### Example 2: POST with JSON Body

**User**: "Call https://example.com/api/order with POST body {\"flightId\":\"FL001\"} and pay if needed"
**Agent**:
1. Identify intent: x402 fetch with POST method and JSON body
2. Verify `mcp_token` is valid
3. Call `dex_tx_x402_fetch` with `url`, `method: "POST"`, `body: "{\"flightId\":\"FL001\"}"`, `headers: "{\"Content-Type\":\"application/json\"}"`
4. Tool handles 402 → sign → retry automatically
5. Display final result to user

### Example 3: User Pastes URL Without Explicit Pay Intent (Edge Case)

**User**: "https://example.com/api/premium-data"
**Agent**:
1. Identify intent: URL provided but no explicit payment intent
2. Ask user: "This URL may require payment (HTTP 402). Would you like me to proceed and pay automatically if the server requires it?"
3. On user confirmation → proceed with `dex_tx_x402_fetch`
4. On user decline → do not call the tool

### Example 4: No Wallet Address for Required Chain (Boundary Case)

**User**: "Pay for https://example.com/api/sol-resource"
**Agent**:
1. Identify intent: x402 payment
2. Call `dex_tx_x402_fetch`
3. Tool returns error: "未获取到 EVM 或 Solana 地址"
4. Inform user: "Your wallet doesn't have a Solana address bound. Please bind a Solana address first to pay for this resource."
